  • 2022-3-29，react18正式发布。

二、新功能

2.1 自动批量更新

   在18之前，只支持 React 事件处理程序内的更新；现在支持 Promise、setTimeout、native event handlers 等，（个人理解：相当于把unstable_batchedUpdates 改为了默认行为，再也不用到处unstable_batchedUpdates了），当然，如果不想自动批量更新，react还留了API，可以通过flushSync这个方法，单独去更新state。

2.2 过渡（Transitions）

Transitions 是 介于紧急更新和非紧急更新中间的一个新概念。

  • 紧急更新：最直击的交互，点击按钮、输入内容等
  • 过渡 更新 ：从一个UI切换到另一个UI的中间态

  非hook环境下可以使用 React.startTransiton(scope)，scope 是一个回调函数，里面的更新任务都会被标记成过渡更新任务，过渡更新任务在渲染并发场景下，会被降级更新优先级，中断更新。

2.3 Suspense

  声明性地指定组件树中尚未准备好显示的部分的加载状态: < Suspense callback={spin}>。

2.5 新的严格模式

官网中说：

在未来，我们希望添加一个特性，允许 React 添加和删除 UI 的部分，同时保留状态。例如，当用户选项卡远离屏幕并返回时，React 应该能够立即显示前一个屏幕。为此，React 将使用与前面相同的组件状态卸载和重新挂载树。

翻译一下是这样的：

  react 未来想要增加一种能力：组件卸载但却保留状态的能力（卸不干净），现在先发布了个严格模式的检查，这个模式一旦开启，会在开发环境模拟组件的挂载和卸载（以检查潜在的问题）。为什么要搞这个？举了个选项卡的例子，当用户切到选项卡之外的时候，就把组件“卸载”掉（状态没丢，只是DOM没了），再切回来的时候，数据还在, 虽然对于开发者它卸不干净，但是从用户的角度来看，它们呈现的效果是相同的。（目的就是体验更好）。
  目前如果要做到它上面说的这个卸载组件但保留状态的能力，八成得依赖外部的能力，比如redux或者浏览器缓存。

PS：严格模式检查仅在开发模式下运行，不会影响生产构建。

一、背景介绍

  • Jest 是一个 JavaScript 测试框架，旨在确保任何 JavaScript 代码的正确性。它提供了易于理解、熟悉且功能丰富的 API 来编写测试用例，并快速地反馈结果。
  • Jest 拥有良好的文档，只需很少的配置，并能 根据你的需求进行扩展。
  • Jest是 Facebook 出的一个测试框架，相对其他测试框架，其一大特点就是就是内置了常用的测试工具，比如自带断言、测试覆盖率工具，实现了开箱即用。

二、主要特点

  • 零配置

    Jest 的目标是在大部分 JavaScript 项目上实现开箱即用， 无需配置。（基于jsmine，所以支持常用 expect、toBe、it等断言写法；内置了测试覆盖率工具istanbul，可以直接在命令行中添加 –coverage 参数开开启此功能）

  • 快照

    能够轻松追踪大型对象的测试。 快照可以与测试代码放在一起，也可以集成进代码 行内。（比如对 React 等常见框架的自动测试）

  • 隔离

    测试程序拥有自己独立的进程 以最大限度地提高性能。（并行执行）

  • 丰富的api

1、问题：Node不适用于计算密集型业务的原因？

回答：对于事件驱动编程来说，如果某个事件的回调函数是计算密集型（CPU被占用）函数，那么这个回调函数将会阻塞所有回调函数的执行。这也是Node不适用于计算密集型业务的原因

2、问题：同样是更新（Update）操作，PUT和PATCH有什么区别呢？

回答：PUT是幂等的，即多次进行PUT操作后的资源总是相同的，因为PUT会更新整个资源；而PATCH是非幂等的，即多次进行PATCH操作会导致资源有不同的变化，因为PATCH只更新资源的部分字段。

1、概念

1.1 编译时

编译时顾名思义就是正在编译的时候.那啥叫编译呢?就是编译器帮你把源代码翻译成机器能识别的代码.(当然只是一般意义上这么说,实际上可能只是翻译成某个中间状态的语言.比如Java只有JVM识别的字节码,C#中只有CLR能识别的MSIL.另外还有啥链接器.汇编器.为了了便于理解我可以统称为编译器)

那编译时就是简单的作一些翻译工作,比如检查老兄你有没有粗心写错啥关键字了啊.有啥词法分析,语法分析之类的过程.就像个老师检查学生的作文中有没有错别字和病句一样.如果发现啥错误编译器就告诉你.如果你用微软的VS的话,点下build.那就开始编译,如果下面有errors或者warning信息,那都是编译器检查出来的.所谓这时的错误就叫编译时错误,这个过程中做的啥类型检查也就叫编译时类型检查,或静态类型检查(所谓静态嘛就是没把真把代码放内存中运行起来,而只是把代码当作文本来扫描下).所以有时一些人说编译时还分配内存啥的肯定是错误的说法.

1.2 运行时

所谓运行时就是代码跑起来了.被装载到内存中去了.(你的代码保存在磁盘上没装入内存之前是个死家伙.只有跑到内存中才变成活的).而运行时类型检查就与前面讲的编译时类型检查(或者静态类型检查)不一样.不是简单的扫描代码.而是在内存中做些操作,做些判断.

2、名词

  • CJS——CommonJS
  • SSR——Server Side Rendering， 服务端渲染
  • CSR——Client Side Rendering，客户端渲染
  • SPR——Serverless Pre-Rendering，有缓存机制的 SSR
  • SSG——Static Site Generation，在编译时运行的 SSR
  • JIT——Just In Time，即时编译，代码在宿主环境执行并编译
  • AOT——Ahead Of Time，预编译代码给宿主环境执行。

  • 前端三剑客，从 Dreamweaver、Fireworks、Flash 到 HTML、CSS、JS 到 View Framework、Node.js CLI、Node.js Framework，再到 Meta Framework、Frontend Paas、Low Code，目前是第四代
  • 低码化，包含两个方向，一是和研发工具结合，实现项目配置的 GUI 可视化操作，二是低码搭建

1、知识扫盲

1.1 什么是卡路里和大卡?

卡路里是热量单位，就是将一克水的温度升高摄氏一度（°C）所需要的热量。1卡路里等于4.1868焦耳。卡路里又被称为小卡。将一千克水加热１°C的热量为一千卡路里，又称为千卡或大卡。
卡路里和我们的日常生活息息相关。一个人，即使什么也不做，每天也要消耗一定数量的卡路里，我们称之为基础代谢。美国女性的平均BMR为1,493卡路里，美国男性的平均BMR为1,662卡路里。食物中的卡路里以热量的形式提供能量，使我们的身体能够发挥作用。我们的身体储存并“燃烧”卡路里作为燃料。

1.2 什么是BMR?

BMR - 基础代谢率。 BMR是人体保持身体静止功能所需的能量（通常以卡路里表示）。 在这种状态下，能量仅用于重要器官（如心脏，肺）的运行，以保持身体功能，（如呼吸，血液循环，控制体温）。 基础代谢率通常占您每天燃烧的卡路里的约60-75％。

interface 和 type的区别

发表于 2021-10-31 更新于 2022-08-28 分类于 前端基础 ， 语言 ， typeScript

ts 的 interface 和 type总是容易混淆， 这里记一下区别。

  1. 功能相同点：都可以描述对象或者函数
  2. 功能不同点：
  • interface 特有的：
  • type 特有的：
1.type 可以声明基本类型别名 或 联合类型 或 元组等类型
    type age = number //基本类型别名，
    type  WW = X | O  // 联合类型         
    type PersonList = [Man, Women] // 元组
     (因为写法上 interface 是没有 ‘=’ 的嘛，所以定义上面这种就不方便)    
2.type 语句中可以使用 typeof 获取实例的 类型进行赋值
    let div = document.createElement('div');
    type B = typeof div
3、其他             
    type click<T> = (data: T) => void;// 用泛型作为类型的入参，描述一个函数
    const handleClickTest: click <string> = (data: string) => {}
    type Foo<T> = [T, T];  // 用泛型作为类型入参，描述一个只有两个元素的数组
    type Tree<T> = T | { a: Tree<T>, b: Tree<T> };
    type StringOrNum = string | number;  
    type StringOrObj = string | { name: string };  
    type Dictionaries = Dictionary<string, Books>;
  1. 写法上的注意点
    • type定义 用 = 号。
    • interface 可以 点的出来，也就是必须照着它描述的样子写函数
    • 函数的描述在名字后面用 ：， 可以使用type 或 interface；
    • 箭头函数返回值的描述， 在箭头函数的() 后面
    • type 和 interface 可以交叉继承
